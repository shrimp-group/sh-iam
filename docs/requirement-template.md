# 业务需求拆解模板

本文档提供业务需求拆解的标准模板与填写示例，所有需求必须按此模板拆解后方可进入开发。

---

## 需求描述模板

```markdown
## 需求：[需求名称]

### 背景
[为什么需要这个需求，解决什么问题]

### 目标
[需求要达成什么效果，可量化的指标]

### 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

### 影响范围
- 涉及模块：[列出模块]
- 涉及接口：[列出 API]
- 涉及数据表：[列出表名]

### 任务拆解

| 任务 | 描述 | 预估 | 优先级 |
|------|------|------|--------|
| T1   | ...  | ...  | P0     |
| T2   | ...  | ...  | P1     |
| T3   | ...  | ...  | P2     |

### 技术方案
[简要技术实现思路，包含关键设计决策]

### 风险点
- [风险1及应对方案]
- [风险2及应对方案]
```

## 填写示例

### 需求：用户登录功能

#### 背景

当前系统仅支持管理员后台登录，普通用户无法自主登录系统。随着业务发展，需要为普通用户提供独立的登录入口，支持多种登录方式，提升用户体验和系统安全性。

#### 目标

1. 支持用户通过手机号 + 验证码登录
2. 支持用户通过用户名 + 密码登录
3. 登录成功后颁发 JWT Token，有效期 2 小时
4. 支持刷新 Token 机制，有效期 7 天

#### 验收标准

- [ ] 用户可通过手机号 + 验证码成功登录，返回有效 JWT Token
- [ ] 用户可通过用户名 + 密码成功登录，返回有效 JWT Token
- [ ] 密码错误时返回明确错误提示，连续错误 5 次锁定账号 30 分钟
- [ ] 验证码 5 分钟内有效，同一手机号 1 分钟内只能发送 1 次
- [ ] Token 过期后可通过刷新 Token 获取新 Token
- [ ] 单元测试覆盖率 >= 80%

#### 影响范围

- 涉及模块：用户模块（user）、认证模块（auth）
- 涉及接口：`POST /api/auth/login`、`POST /api/auth/sms-code`、`POST /api/auth/refresh`
- 涉及数据表：`user`、`user_login_log`

#### 任务拆解

| 任务 | 描述                   | 预估   | 优先级 |
|----|----------------------|------|-----|
| T1 | 设计并实现用户登录接口（用户名+密码）  | 1天   | P0  |
| T2 | 实现手机验证码发送与校验逻辑       | 1天   | P0  |
| T3 | 实现 JWT Token 颁发与刷新机制 | 1天   | P0  |
| T4 | 实现登录失败锁定机制           | 0.5天 | P1  |
| T5 | 编写单元测试与集成测试          | 1天   | P1  |
| T6 | 登录日志记录与查询            | 0.5天 | P2  |

#### 技术方案

1. **认证流程**：采用 JWT 无状态认证方案，Access Token 有效期 2 小时，Refresh Token 有效期 7 天
2. **验证码**：使用 Redis 存储验证码，设置 5 分钟过期；发送频率限制通过 Redis 计数器实现
3. **密码加密**：使用 BCrypt 算法，salt rounds = 10
4. **锁定机制**：登录失败次数存储于 Redis，key 为 `login_fail:{userId}`，5 次失败后设置 30 分钟锁定
5. **接口设计**：RESTful 风格，统一响应格式 `{ code, message, data }`

#### 风险点

- **短信服务不可用**：接入短信服务商备用通道，降级为仅支持密码登录
- **Redis 宕机导致验证码丢失**：验证码同时写入数据库，Redis 不可用时降级查询数据库
- **暴力破解风险**：实施 IP 级别限流（单 IP 每分钟最多 10 次请求），结合账号锁定机制双重防护
