# 技术债务目录规范

## 目录结构
```
docs/tech-debts/
├── INDEX.md                    # 债务索引（汇总所有债务状态）
├── 性能风险/
│   └── 001-高并发瓶颈.md
├── 内存隐患/
│   └── 002-内存泄漏.md
├── 线程与并发风险/
│   └── 003-死锁风险.md
├── 安全风险/
│   └── 004-SQL注入.md
└── 其他潜在风险/
    └── 005-可观测性缺失.md
```

## 文件命名规范

- 格式：`{序号}-{中文简述}.md`
- 序号：3 位数字，从 001 开始递增
- 分类通过子目录体现，不再在文件名中重复分类简称
- 中文简述：2-6 个中文字描述
- 示例：`性能风险/001-慢查询.md`、`安全风险/005-SQL注入.md`

## 分类目录说明

| 目录名     | 原分类代码   | 子类说明            |
|---------|---------|-----------------|
| 性能风险    | `perf`  | 性能瓶颈、慢查询、高并发等   |
| 内存隐患    | `mem`   | 内存泄漏、OOM、缓存膨胀等  |
| 线程与并发风险 | `conc`  | 死锁、竞态条件、线程安全等   |
| 安全风险    | `sec`   | SQL注入、XSS、权限漏洞等 |
| 其他潜在风险  | `other` | 可观测性缺失、配置硬编码等   |

## 状态流转规则

### 状态定义

| 状态          | 含义         | 颜色标记 |
|-------------|------------|------|
| open        | 待处理，尚未开始解决 | 🔴   |
| in-progress | 正在处理中      | 🟡   |
| resolved    | 已解决        | 🟢   |

### 状态流转
```
open → in-progress → resolved
```
- open → in-progress：开始处理债务时更新
- in-progress → resolved：债务解决后更新
- 不允许回退状态（resolved 不可变为 open 或 in-progress）
- 如果已解决的债务再次出现，创建新的债务记录

### 状态更新操作
1. 修改债务文件中的"状态"字段
2. 填写"解决日期"（resolved 时）
3. 填写"当前解决方案"或"建议解决方案"
4. 更新 INDEX.md 中的对应条目

## 新增债务流程
1. 确定下一个可用序号（查看 INDEX.md 中最大序号 +1）
2. 使用 tech-debt.md.template 模板创建债务文件，放入对应分类子目录（如 `性能风险/`、`安全风险/` 等）
3. 填写所有必填字段
4. 更新 INDEX.md，添加新条目到 Open 分组，路径需包含子目录（如 `性能风险/001-慢查询.md`）
5. 更新统计概览数据

## 解决债务流程
1. 修改债务文件：状态改为 resolved，填写解决日期和解决方案
2. 更新 INDEX.md：将条目从 Open/In Progress 移到 Resolved 分组，路径需包含子目录（如 `安全风险/005-SQL注入.md`）
3. 更新统计概览数据
